# Page patterns

## Executive summary

Use 3–4 blocks. Each block has a label, a conclusion sentence and 1–2 supporting points. The page should answer the deck's main question.

## SCQA page

Four columns or stacked blocks: Situation, Complication, Question, Answer. Keep it concise; the answer links to the deck's recommendation.

## One-chart page

Left or center chart, right rail for so-what, caveat and decision implication. The chart must prove the action title.

## 2x2 matrix

Use for prioritization or portfolio choices. Preferred pattern: left 2x2 + right insight rail with implications for each quadrant.

## Roadmap

Use workstreams as rows and periods as columns. Show waves, milestones and decision gates. Avoid more than 4–5 periods on one page.

## Waterfall

Use for TCO, savings, revenue bridge and margin bridge. Validate that start + deltas = end.

## Heatmap

Use for maturity, control gaps, risk by function or capability readiness. Keep color scale simple and explain the meaning.

## Option evaluation table

Rows are options, columns are criteria. Show weights where useful. Highlight the preferred option and connect the result to the recommendation.

## RACI matrix

Use for governance, operating model and implementation roles. Keep roles to 4–6 and activities to 5–8 where possible.

## Risk matrix

Use likelihood × impact. Include mitigation priorities next to the chart, not only dots.

## Architecture page

Use layers: experience, orchestration, knowledge/data, model/platform and control/governance. Avoid mixing business process and infrastructure on the same visual plane.
